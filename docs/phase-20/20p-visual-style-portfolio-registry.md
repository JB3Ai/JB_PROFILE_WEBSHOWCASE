## Purpose

The public website now needs a stronger premium founder-platform frame before more apps, products, systems, or cutsheets are added.

20K through 20O created a clean IsikoloAI asset, image review path, cutsheet planning layer, and app-card copy decision. The new strategic context is broader: JB³Ai is not a single-app portfolio. It is a founder-led operating architecture that spans OS³ Dash, IsikoloAI, VoiceGrid Ai, ViewGridAi, InvestigatorAi, ClipboardAi, Newsroom, Forensic Intelligence Systems, and the JB³Ai Super Agent Operating System.

20P therefore pauses visible app-card integration and defines the visual direction, portfolio categories, public-safe descriptions, source-material controls, and next milestone sequence.

## Visual Direction Decision

The design target is:

- Premium founder operating system.
- Editorial technology platform.
- Structured intelligence.
- Precise, chic, human-led.
- Less generic AI SaaS.
- Less rounded card softness.
- Less neon/glow dependency.
- More architectural, restrained, premium, and image-led.

The site should feel like a command architecture with human judgment at the center, not a generic dashboard full of disconnected app tiles.

## Visual Style Controls

| Design area | Current risk | Required control | Future implementation note |
| --- | --- | --- | --- |
| Card radius | Too many soft rounded SaaS cards can make the portfolio feel generic. | Use fewer, larger architectural panels with restrained radius. | Reserve softer cards for secondary lists only. |
| Glow effects | Neon/glow dependency can make the site feel like a template. | Use glow as atmosphere, not as the primary design language. | Prefer shadow, material depth, editorial contrast, and image-led hierarchy. |
| Background gradients | Repeated gradients can blur section identity. | Keep gradients subtle and directional. | Use each portfolio layer to carry a distinct tonal field. |
| Typography | Product tiles may become too similar and lose founder-platform authority. | Use editorial headings, precise labels, and confident short descriptions. | Avoid overloading cards with buzzwords. |
| Spacing | Dense app grids can feel like a directory. | Use generous vertical rhythm and sectional separation. | Let platform layers breathe before app cards appear. |
| Image treatment | Unreviewed images can imply unsupported maturity or active services. | Use approved production assets only, with contained crop rules. | Each image needs alt text, usage mapping, and mobile crop QA. |
| Icon use | Generic AI icons can flatten the brand. | Use icons sparingly and only as navigation aids. | Favor typography, diagrams, and product imagery over icon packs. |
| Dark-mode contrast | Dark panels can become low-contrast and hard to scan. | Maintain readable contrast and clear hierarchy. | Test every portfolio card at desktop and 390px mobile. |
| Card hierarchy | Random app cards can imply equal readiness. | Separate platform, public apps, forensic systems, and resources. | Readiness and risk labels should guide placement. |
| Mobile density | Portfolio sections can become cramped on 390px screens. | Limit first mobile pass to a small curated set. | Batch integration should be visual-QA gated. |
