## Product-by-Product Purpose-Made Visual Briefs

### A. OS3 Dash

Product positioning:

Founder OS / Operating Dashboard.

Visual objective:

Communicate structured command, daily visibility, approvals, workflow clarity, and execution discipline without exposing any private OS material.

Scene / composition brief:

A staged command-dashboard interface displayed as a premium dark steel panel. The composition should show a clean hierarchy: priority lane, approval lane, signal/alert lane, and execution flow lane. Labels must be generic and minimal.

Suggested style:

Premium founder OS, editorial technology platform, controlled dashboard architecture, sharp panels, low glow, no generic SaaS cartoon style.

Colour palette guidance:

Deep navy-black background, slate blue steel panels, chrome dividers, subtle luminous tech green accents.

Typography guidance, if any:

Minimal uppercase micro-labels only, with generic words such as Priority, Approval, Signal, Workflow, and Review.

Must include:

- Structured command dashboard feel.
- Generic workflow panels.
- Subtle signal lines or status indicators.
- Card-safe focal area.

Must avoid:

- Real inbox, calendar, notes, Gmail, Drive, legal, family, financial, client, or credential data.
- Private OS routes.
- Real names, emails, task titles, account labels, or figures.
- Security, uptime, scale, or SaaS maturity claims.

Safety controls:

Use staged/abstract data only. Visible text must be generic and reviewed at desktop and 390px mobile sizes.

Proposed filename:

`os3-dash-command-dashboard.webp`

Proposed registry key:

`os3DashCommandDashboard`

Proposed alt text:

OS3 Dash command dashboard concept showing structured workflow visibility.

Crop guidance:

16:10 preferred, 4:3 acceptable. Keep one clean central dashboard plate readable on desktop and mobile. Avoid dense text and edge-critical labels.

Future generation notes:

Generate or design as an owned conceptual interface, not a screenshot of the real OS.

Future implementation notes:

Requires 20W5 prompt approval, generation/review milestone, optimisation milestone, registry milestone, and later card-render milestone.

### B. JB3Ai Super Agent OS

Product positioning:

AI Workforce Architecture.

Visual objective:

Show JB3Ai Super Agent OS as a managed AI workforce architecture with human oversight and structured routing, not a chatbot directory.

Scene / composition brief:

An architecture plate with Adam as the executive command layer at the top or centre, connected to 6-8 grouped specialist workflow clusters. The visual should feel like a command map, not an org chart stuffed with names.

Suggested style:

Luxury intelligence architecture, editorial systems map, clean routing lines, controlled depth, high-signal structure.

Colour palette guidance:

Deep navy-black, slate blue steel, chrome strokes, restrained green highlights for active routing paths.

Typography guidance, if any:

Use broad group labels such as Operations, Research, Development, Creative, Growth, Legal Support, Finance Support, Voice, and Wellbeing only if legible. Avoid dense agent roster text.

Must include:

- Adam / command layer.
- Specialist workflow groups.
- Human oversight or approval cue.
- Routing/coordination feel.

Must avoid:

- Autonomous replacement implication.
- Legal or finance authority claims.
- Internal architecture secrets.
- Private workflow labels.
- Chatbot-directory grid.

Safety controls:

Frame as coordination and structured execution. Do not imply agents make final decisions without human review.

Proposed filename:

`jb3ai-super-agent-os-architecture.webp`

Proposed registry key:

`jb3aiSuperAgentOsArchitecture`

Proposed alt text:

JB3Ai Super Agent OS architecture concept showing coordinated specialist workflows.

Crop guidance:

16:10 or 3:2. Keep command layer and major groups readable. Avoid over-dense text that fails on 390px mobile.

Future generation notes:

Use a purpose-made systems map with owned styling. Do not reuse internal diagrams if they contain private workflow detail.

Future implementation notes:

Can become a flagship Batch 01 card visual only after human review confirms no autonomy, legal/finance, or private-workflow overclaim.
